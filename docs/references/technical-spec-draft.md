# Драфт ТЗ: HiveMonitor

## 1. Назначение проекта

HiveMonitor - рабочее название комплексной системы мониторинга ульев и пасеки, включающей IoT-устройства на базе ESP, backend, базу данных, web-интерфейс и мобильные приложения для Android и iOS.

Система должна собирать данные с ульев, сохранять историю измерений, отображать состояние пасеки пользователю, отправлять уведомления о важных событиях и помогать анализировать изменения веса, температуры, влажности и факта открытия улья.

Прошивка ESP является только одним из компонентов системы. Полное решение должно включать серверную часть, пользовательские интерфейсы, управление устройствами, хранение данных, авторизацию и систему уведомлений.

## 2. Цели системы

- Автоматически собирать данные с одного или нескольких ульев.
- Отображать актуальное состояние каждого улья.
- Хранить историю измерений.
- Показывать графики веса, температуры и влажности.
- Фиксировать события открытия улья.
- Выявлять резкие изменения веса и другие подозрительные события.
- Отправлять уведомления пользователю.
- Поддерживать работу с несколькими пасеками и несколькими ульями.
- Давать пользователю доступ через web-интерфейс и мобильные приложения.
- Поддерживать интерфейсы на русском, украинском и английском языках.

## 3. Основные компоненты

### 3.1. IoT-устройство

Устройство устанавливается на улей и отвечает за сбор данных с датчиков.

Основные функции:

- измерение веса улья;
- измерение температуры;
- измерение влажности;
- определение факта открытия улья;
- измерение уровня батареи/питания в процентах и напряжения в вольтах;
- отправка данных на сервер;
- локальное сохранение данных при отсутствии связи;
- экономия питания через режим сна.

### 3.2. Backend

Backend принимает данные от устройств, сохраняет их, предоставляет API для web и мобильных приложений, управляет пользователями, пасеками, ульями, устройствами и уведомлениями.

Основной backend-стек:

- Go;
- PostgreSQL;
- MQTT broker.

Развертывание системы проектируется по подходу Docker-first. Основной ожидаемый вариант размещения - VPS или выделенный сервер, но архитектура не должна быть жестко привязана к конкретному провайдеру.

Основные функции:

- прием телеметрии от устройств;
- проверка и нормализация данных;
- хранение истории измерений;
- управление пользователями;
- управление пасеками и ульями;
- управление устройствами;
- управление конфигурациями устройств;
- обработка событий и тревог;
- отправка push-уведомлений;
- предоставление API для web и mobile.

## 6. IoT-устройство и прошивка

### 6.1. Аппаратная часть

Предварительная аппаратная платформа:

- совместимые платы ESP8266 и ESP32;
- датчик веса через HX711;
- настраиваемый датчик температуры и влажности: DHT22 по умолчанию, с возможностью выбора других типов;
- датчик Холла для определения открытия улья;
- RTC-модуль как опциональный источник времени;
- измерение уровня батареи/питания в процентах и напряжения в вольтах;
- Wi-Fi для передачи данных;
- Bluetooth, если выбранный контроллер поддерживает его;
- локальная память для хранения настроек и буферизации данных.

Проект должен предусматривать две альтернативные прошивки:

- прошивка для ESP8266;
- прошивка для ESP32.

Конкретные модели плат на этапе драфта ТЗ не фиксируются. Выбор точных плат ESP8266/ESP32 должен быть сделан на этапе аппаратного прототипирования.

Общая бизнес-логика устройства должна быть максимально общей, но аппаратно-зависимые части должны учитывать различия ESP8266 и ESP32: режимы сна, пробуждение, Bluetooth, доступную память и работу с GPIO.

Прошивка должна быть переработана из одного `.ino`-файла в модульную структуру проекта. Общий код должен быть вынесен в shared-модули, а платформенно-зависимый код - в отдельные директории ESP8266 и ESP32.

Основной инструмент сборки прошивки - PlatformIO. Для разработки прошивок используется Arduino framework внутри PlatformIO, если не будет принято отдельное решение о переходе на другой framework.

PlatformIO должен поддерживать отдельные окружения сборки:

- `esp8266`;
- `esp32`.

Предварительная структура прошивки:

```text
firmware/
  platformio.ini
  shared/
    sensors/
    communication/
    storage/
    time/
    config/
    web/
    update/
    domain/
  esp8266/
    src/
    platform/
  esp32/
    src/
    platform/
```

Примеры модулей:

- `sensors` - датчик веса, DHT22, датчик открытия улья, будущие датчики;
- `communication` - Wi-Fi, MQTT, HTTP/API;
- `storage` - настройки, LittleFS-буфер измерений, служебные флаги;
- `time` - RTC, NTP, выбор источника времени;
- `config` - конфигурация устройства и синхронизация настроек;
- `web` - локальный web-интерфейс устройства;
- `update` - локальное обновление прошивки и будущие OTA-сценарии;
- `domain` - бизнес-логика измерений, событий и проверки ошибок.

Предварительное назначение пинов по умолчанию:

| Назначение                                | Пин ESP8266 |
| ----------------------------------------- | ----------- |
| HX711 DOUT                                | D2          |
| HX711 SCK                                 | D3          |
| Датчик температуры/влажности по умолчанию | D4          |
| Датчик открытия улья                      | D5          |

Пины датчиков должны быть настраиваемыми через конфигурацию устройства везде, где это технически возможно для выбранной платы и конкретного модуля. Значения из таблицы являются значениями по умолчанию, а не жестко зашитой схемой.

Для MVP изменение пинов датчиков выполняется через локальный web-интерфейс устройства. Удаленное изменение пинов через backend, web-интерфейс системы и мобильные приложения переносится на этап после MVP.

### 6.2. Основной сценарий устройства

1. Устройство просыпается по таймеру или внешнему событию.
2. Считывает данные с датчиков.
3. Проверяет корректность измерений.
4. Подключается к Wi-Fi.
5. Отправляет данные на сервер.
6. При ошибке связи сохраняет данные локально.
7. После успешной отправки уходит в режим сна.

Период планового измерения по умолчанию - 30 минут. Период должен быть настраиваемым параметром устройства.

### 6.3. Логика измерения веса

Для работы с модулем HX711 выбирается библиотека `GyverHX711`.

Пины HX711 `DOUT` и `SCK` должны быть настраиваемыми. Значения по умолчанию: `DOUT = D2`, `SCK = D3`.

Если улей не был открыт и вес изменился больше `5.0 кг`, устройство должно перепроверить вес несколькими измерениями.

Пороги должны быть настройками устройства, а не жестко зашитыми константами. Значения по умолчанию:

- `WEIGHT_THRESHOLD = 5.0 кг` - порог изменения веса, после которого требуется дополнительная проверка;
- `SIGNIFICANT_WEIGHT_CHANGE = 20.0 кг` - допустимое значительное изменение веса после открытия улья, которое не считается ошибкой.

Пороги должны изменяться:

- через локальный web-интерфейс устройства;
- через backend;
- через web-интерфейс системы;
- через мобильные приложения.

Предварительная логика:

1. Выполнить три измерения веса.
2. Между измерениями сделать паузу 100 мс.
3. Если разница между соседними измерениями меньше заданного порога, считать измерение стабильным.
4. Если измерение стабильное, использовать среднее значение.
5. Если измерение нестабильное, пометить данные как ошибочные.
6. Если между предыдущим и текущим измерением был зафиксирован факт открытия улья, значительное изменение веса в пределах `SIGNIFICANT_WEIGHT_CHANGE` не считается ошибкой.

В старых заметках также упоминается, что при открытии улья изменение веса потенциально может достигать 50 кг. Текущим рабочим порогом для логики считается `20.0 кг`.

### 6.4. Калибровка весового модуля

Весовой модуль должен поддерживать калибровку как локально через web-интерфейс устройства, так и удаленно через backend.

Для MVP обязательна калибровка через локальный web-интерфейс устройства. Калибровка через backend переносится на этап после MVP.

В MVP запускать калибровку может только пользователь, авторизованный в локальном web-интерфейсе устройства.

Калибровка через web-интерфейс устройства:

- запуск процесса тарирования;
- установка нулевого значения без нагрузки;
- ввод эталонного веса;
- измерение эталонного веса;
- расчет калибровочного коэффициента;
- сохранение коэффициента в локальную память устройства;
- проверочное измерение после калибровки;
- сброс калибровки к заводским или предыдущим значениям.

Калибровка через backend:

- просмотр текущих калибровочных параметров устройства;
- отправка команды тарирования;
- отправка эталонного веса;
- запуск удаленного измерения для калибровки;
- сохранение рассчитанного коэффициента на устройстве;
- хранение копии калибровочных параметров на сервере;
- отображение результата калибровки пользователю;
- журналирование истории калибровок.

Калибровочные данные должны включать:

- коэффициент калибровки;
- значение тары;
- дату и время калибровки;
- способ калибровки: локально или через backend;
- пользователя, если калибровка запускалась через сервер;
- версию прошивки;
- комментарий или описание эталонного груза, если задано.

Требуется уточнить:

- в каких единицах система хранит вес;
- какой минимальный и максимальный допустимый эталонный вес;
- можно ли выполнять калибровку без подключения к backend;
- как синхронизировать локальные изменения калибровки с сервером;
- нужно ли запрещать обычные измерения во время калибровки;
- нужен ли механизм отката к предыдущей калибровке.

### 6.5. Энергосбережение

Устройство должно большую часть времени находиться в режиме сна.

Для ESP8266 должна поддерживаться настройка схемы пробуждения по датчику открытия улья:

- `HALL_WAKE_RST` - датчик открытия подключен через внешнюю схему к `RST`, открытие улья может будить устройство из deep sleep;
- `HALL_NO_WAKE` - отдельной схемы пробуждения нет, открытие улья проверяется только при плановом пробуждении устройства.

Логика для режима `HALL_WAKE_RST`:

1. Устройство может проснуться от планового таймера или от события открытия улья.
2. После пробуждения устройство проверяет датчик открытия.
3. Если открытие зафиксировано, устройство сохраняет факт открытия и время события.
4. Устройство отправляет событие открытия на сервер при наличии связи.
5. При следующем измерении веса значительное изменение веса не считается ошибкой, если ранее был установлен флаг открытия.

Логика для режима `HALL_NO_WAKE`:

1. Устройство просыпается только по плановому таймеру.
2. Состояние датчика открытия проверяется во время планового цикла измерения.
3. Если открытие было кратковременным между циклами сна и схема не будит устройство, событие может быть не зафиксировано.
4. Этот режим считается упрощенным и менее точным для фиксации открытия улья.

Для ESP32 нужно предусмотреть отдельную реализацию пробуждения по GPIO средствами deep sleep wake-up, если выбранные GPIO и схема подключения это позволяют.

Требуется уточнить:

- период планового измерения;
- способ пробуждения;
- должен ли датчик открытия будить устройство немедленно;
- используется ли GPIO16-RST для пробуждения ESP8266;
- требуется ли внешний таймер.

### 6.6. Источник времени

Устройство должно поддерживать два варианта получения времени:

- NTP-синхронизация через Wi-Fi;
- внешний RTC-модуль, если он установлен и включен в настройках.

Источник времени должен быть настраиваемым:

- `NTP` - использовать синхронизацию времени через сеть;
- `RTC` - использовать внешний RTC-модуль;
- `AUTO` - использовать RTC, если он доступен, иначе NTP.

Если RTC-модуль установлен и включен, устройство должно использовать его как основной источник времени. Это должно уменьшить сетевой трафик и энергозатраты, потому что устройству не нужно выполнять NTP-синхронизацию при каждом пробуждении.

При использовании RTC устройство все равно должно периодически синхронизировать RTC по NTP, но значительно реже, чем в режиме без RTC. Интервал синхронизации RTC должен быть отдельной настройкой устройства.

Если RTC-модуль отсутствует, отключен или выдает некорректное время, устройство должно использовать NTP при наличии Wi-Fi.

Требуется уточнить:

- какая модель RTC-модуля будет использоваться;
- как определять, что RTC установлен и исправен;
- как часто синхронизировать RTC по NTP;
- что делать, если нет ни RTC, ни доступа к NTP;
- нужно ли хранить время последней успешной синхронизации;
- как учитывать часовой пояс пасеки.

### 6.7. Датчики температуры и влажности

Модуль температуры и влажности должен быть настраиваемым.

Для MVP датчиком по умолчанию считается DHT22, но прошивка и конфигурация должны предусматривать выбор типа датчика и пинов подключения.

Требуется поддержать сценарии:

- один комбинированный датчик температуры и влажности;
- отдельный датчик температуры;
- отдельный датчик влажности;
- отключение измерения температуры;
- отключение измерения влажности.

Настраиваемые параметры:

- тип датчика температуры/влажности;
- пин датчика температуры/влажности;
- тип отдельного датчика температуры;
- пин отдельного датчика температуры;
- тип отдельного датчика влажности;
- пин отдельного датчика влажности;
- включение и отключение каждого датчика;
- периодичность измерения, если она отличается от общего цикла устройства.

Система должна хранить в конфигурации, какие датчики установлены и какие значения устройство должно отправлять в телеметрии.

### 6.8. Локальный web-интерфейс устройства

IoT-устройство должно иметь локальный web-интерфейс для первичной настройки, диагностики и управления модулями.

Локальный web-интерфейс должен быть доступен постоянно в рабочей Wi-Fi сети устройства. Режим временной точки доступа используется для первичной настройки, восстановления доступа или сброса сетевых параметров.

Доступ к локальному web-интерфейсу должен быть защищен авторизацией. Минимальное требование - пароль администратора устройства. В локальном web-интерфейсе не требуется разделение на роли: любой авторизованный пользователь получает доступ ко всем функциям локального интерфейса.

Локальный web-интерфейс должен позволять изменять настройки устройства даже без связи с backend. Такие изменения сохраняются локально и синхронизируются с backend после восстановления связи.

Для MVP локальный web-интерфейс должен быть минимальным, аккуратным и функциональным. Он должен закрывать базовые сценарии настройки и обслуживания устройства без лишней визуальной сложности.

После MVP локальный web-интерфейс должен развиваться до полноценного интерфейса устройства: не перегруженного, но визуально и функционально достаточно качественного, чтобы восприниматься как нормальное лицо продукта.

Предварительный сценарий:

1. При первом запуске или после сброса настроек устройство поднимает временную Wi-Fi точку доступа.
2. Пользователь подключается к этой точке доступа с телефона или компьютера.
3. Пользователь открывает локальный web-интерфейс устройства.
4. Пользователь задает или меняет пароль администратора устройства.
5. Через web-интерфейс задает параметры подключения, модулей и передачи данных.
6. Устройство сохраняет настройки в локальную память.
7. После сохранения устройство перезагружается или переключается в рабочий режим.

Функции локального web-интерфейса:

- настройка Wi-Fi: SSID, пароль, режим подключения;
- сканирование доступных Wi-Fi сетей;
- настройка MQTT: сервер, порт, логин, пароль, топик, TLS;
- настройка идентификатора устройства;
- настройка токена или ключа доступа устройства (`deviceToken`);
- настройка источника времени: RTC, NTP или авто;
- настройка пинов датчика веса HX711;
- включение и отключение датчика температуры и влажности;
- выбор типа датчика температуры и влажности;
- настройка пинов датчиков температуры и влажности;
- настройка отдельных датчиков температуры и влажности;
- включение и отключение датчика открытия улья;
- включение и отключение датчика веса;
- включение и отключение RTC-модуля;
- калибровка весового модуля;
- резервирование места под будущие модули и датчики;
- настройка интервала измерений;
- настройка режима сна;
- выбор типа контроллера и схемы пробуждения;
- просмотр текущих показаний датчиков;
- просмотр статуса Wi-Fi и MQTT;
- просмотр версии прошивки;
- локальное обновление прошивки через web-интерфейс;
- просмотр состояния локальной памяти;
- очистка локального буфера данных;
- сброс настроек устройства;
- включение и отключение Bluetooth, если выбранный контроллер его поддерживает.

Настройки модулей должны храниться так, чтобы прошивка могла работать с разными комплектациями устройства. Например, устройство может быть установлено только с весами, без датчика открытия или без датчика температуры и влажности.

Требуется уточнить:

- где хранить настройки: EEPROM/NVS, LittleFS, SPIFFS или другая память;
- как пользователь попадает в режим настройки после первичной установки;
- поддерживается ли Bluetooth выбранным контроллером.

### 6.9. Обновление прошивки

IoT-устройство должно поддерживать обновление прошивки без физического подключения к программатору.

Требуемые способы обновления:

- локальная загрузка файла прошивки через web-интерфейс устройства - обязательный способ для MVP;
- ручное обновление через USB/serial как аварийный способ восстановления - обязательный способ для MVP;
- OTA-обновление по локальной сети - после MVP;
- OTA-обновление через backend или сервер обновлений - после MVP.

Локальное обновление через web-интерфейс:

- пользователь авторизуется и открывает web-интерфейс устройства;
- выбирает файл прошивки;
- устройство проверяет файл перед установкой;
- устройство показывает ход загрузки и установки;
- после успешной установки устройство перезагружается;
- после перезагрузки устройство показывает новую версию прошивки.

OTA-обновление через backend:

- backend хранит доступные версии прошивки;
- устройство может проверять наличие новой версии;
- backend может назначать обновление конкретному устройству или группе устройств;
- устройство скачивает прошивку по защищенному каналу;
- устройство подтверждает успешное обновление после перезагрузки.

Требования к безопасности обновлений:

- проверка версии прошивки;
- проверка целостности файла;
- защита от установки неподходящей прошивки;
- авторизация доступа к обновлению;
- желательно использовать подпись прошивки;
- обновление не должно раскрывать токены и настройки устройства.

Требования к надежности:

- при ошибке загрузки устройство должно остаться на старой прошивке;
- при ошибке проверки файл не должен устанавливаться;
- настройки устройства должны сохраняться после обновления;
- желательно предусмотреть механизм отката или безопасный recovery-режим;
- устройство должно логировать результат обновления локально и отправлять статус на backend.

Требуется уточнить:

- поддерживает ли выбранная плата достаточный объем памяти для OTA;
- нужен ли dual-partition/rollback механизм;
- как хранить версии прошивок на backend;
- нужен ли staged rollout для обновления части устройств;
- можно ли обновлять web-интерфейс устройства отдельно от основной прошивки.

## 7. Передача данных

### 7.1. Канал передачи

Основной канал передачи телеметрии от устройств - MQTT через отдельный MQTT broker.

Использование TLS для MQTT должно быть настраиваемым параметром устройства и сервера. Система должна поддерживать оба режима:

- MQTT без TLS;
- MQTT over TLS.

Для публичного продакшен-развертывания TLS рекомендуется как обязательный режим. Для локальной разработки, тестовых стендов или изолированных сетей допускается MQTT без TLS.

Авторизация устройства выполняется по `deviceId` и `deviceToken`. MQTT broker/backend должны проверять, что устройство существует, активно и имеет право публиковать данные в свои топики.

Структура MQTT-топиков:

```text
hives/{deviceId}/telemetry
hives/{deviceId}/events
hives/{deviceId}/status
hives/{deviceId}/commands
hives/{deviceId}/config
```

Назначение топиков:

- `telemetry` - регулярные измерения устройства;
- `events` - события устройства, например открытие улья или ошибка датчика;
- `status` - статус устройства, online/offline, версия прошивки, состояние модулей;
- `commands` - команды от backend к устройству;
- `config` - передача и синхронизация конфигурации устройства.

Команды от backend к устройству не являются обязательными для MVP, но должны быть предусмотрены архитектурно и относятся к высокому приоритету после MVP.

Примеры будущих команд:

- применить новую конфигурацию;
- выполнить тарирование;
- выполнить калибровку;
- перезагрузить устройство;
- отправить текущий статус;
- очистить локальный буфер;
- выполнить диагностическое измерение.

### 7.2. Формат сообщения

Формат телеметрии на сервер - JSON.

```json
{
  "deviceId": "hive-001",
  "timestamp": "2026-04-28T09:00:00Z",
  "weight": 42.35,
  "weightChange": 1.2,
  "temperature": 24.1,
  "humidity": 61.0,
  "hiveOpened": false,
  "errorFlag": false,
  "batteryPercent": 87,
  "batteryVoltage": 3.92,
  "rssi": null
}
```

Минимальные обязательные поля:

- `deviceId` - идентификатор устройства;
- `timestamp` - время измерения;
- `weight` - текущий вес;
- `weightChange` - разница с предыдущим измерением;
- `temperature` - температура;
- `humidity` - влажность;
- `hiveOpened` - признак открытия улья;
- `errorFlag` - признак ошибочного или подозрительного измерения.
- `batteryPercent` - уровень батареи в процентах, от 0 до 100;
- `batteryVoltage` - напряжение батареи в вольтах.

Дополнительные поля могут добавляться по мере развития устройства, например `rssi`, версия прошивки, статус датчиков и диагностическая информация.

## 8. Backend

Backend реализуется на Go и использует PostgreSQL как основную базу данных. Для приема телеметрии от устройств используется MQTT broker.

Backend, база данных, MQTT broker и связанные сервисы должны запускаться через Docker/Docker Compose. Основной сценарий размещения - VPS или выделенный сервер. При этом проект должен оставаться переносимым и не зависеть от конкретного хостинг-провайдера.

## 11. Конфигурация устройства

Устройство должно поддерживать локальную настройку через собственный web-интерфейс.

Все настройки устройства должны храниться как конфигурация, а не быть жестко зашитыми в прошивку. Конфигурация должна редактироваться несколькими способами:

- через локальный web-интерфейс устройства;
- через backend;
- через web-интерфейс системы;
- через мобильные приложения Android и iOS.

Изменения настроек, сделанные через backend, web-интерфейс системы или мобильное приложение, должны доставляться на устройство и применяться устройством. Изменения, сделанные локально на устройстве, должны синхронизироваться обратно на backend после появления связи.

Устройство должно оставаться настраиваемым локально даже при недоступности backend. Локальные изменения конфигурации должны попадать в очередь синхронизации и отправляться на backend при первой успешной связи.

Требуется хранить версию конфигурации и время последнего изменения, чтобы корректно решать конфликты между локальными и серверными настройками.

Правило разрешения конфликтов конфигурации:

- автоматически применяется более новая версия настройки по времени изменения;
- спорные или потенциально опасные изменения должны логироваться;
- пользователь должен видеть информацию о конфликте в web-интерфейсе системы;
- для критичных параметров допускается требовать ручное подтверждение.

Базовый способ первичной настройки:

- настройка Wi-Fi через временную точку доступа;
- настройка через web-интерфейс устройства;
- настройка через мобильное приложение, если оно умеет находить устройство в режиме настройки;
- прошивка с заранее заданными параметрами;
- QR-код или код привязки устройства.

Для MVP привязка устройства к улью выполняется вручную через web-админку системы по `deviceId`.

Для MVP `deviceId` по умолчанию генерируется устройством из MAC/chip ID. Пользователь должен иметь возможность изменить `deviceId` через локальный web-интерфейс устройства. На сервере устройство регистрируется и привязывается к улью по этому `deviceId`.

Для MVP `deviceToken` генерируется в web-админке системы и вручную вводится пользователем в локальном web-интерфейсе устройства.

Устройство можно перепривязать к другому улью, но система должна сохранять историю привязок. Измерения, полученные до перепривязки, должны оставаться связанными со старым ульем и не должны задним числом переноситься на новый улей.

После MVP должны быть рассмотрены более удобные способы привязки:

- QR-код на устройстве;
- привязка через мобильное приложение;
- автоматическое обнаружение устройства в режиме настройки;
- автоматическая передача `deviceToken` на устройство после успешной привязки.

Через web-интерфейс устройства должны настраиваться:

- параметры Wi-Fi;
- параметры MQTT;
- идентификатор устройства;
- ключ доступа или токен устройства;
- включенные аппаратные модули;
- типы датчиков и назначение пинов;
- интервал измерений;
- параметры сна;
- схема пробуждения датчиком открытия;
- источник времени и параметры NTP/RTC;
- параметры локального хранения;
- калибровка весового модуля;
- пороги веса и правила определения ошибок;
- пороги температуры и влажности;
- правила уведомлений устройства;
- обновление прошивки;
- Bluetooth, если он есть в выбранном контроллере.

Модули должны включаться и отключаться независимо:

- датчик веса;
- датчик температуры и влажности;
- отдельный датчик температуры;
- отдельный датчик влажности;
- датчик открытия улья;
- RTC-модуль;
- Bluetooth-модуль;
- будущие дополнительные датчики и расширения.

Примеры настраиваемых параметров устройства:

- SSID и пароль Wi-Fi;
- адрес, порт и параметры авторизации MQTT;
- режим TLS для MQTT;
- идентификатор устройства;
- токен или ключ доступа (`deviceToken`);
- режим MQTT TLS: включен или выключен;
- включенные датчики и модули;
- типы датчиков температуры/влажности;

- режим сна;

- параметры локального буфера;

- режим Bluetooth;

## 12. Хранение данных

Для MVP сервер должен хранить всю историю измерений без автоматического удаления.

- хранение технических логов.

### 12.1. Локальное хранение на устройстве

Устройство должно разделять хранение небольших настроек и хранение накопленных измерений.

Для ESP8266:

- EEPROM используется только для небольших настроек или служебных флагов, если это необходимо;
- буфер измерений при отсутствии связи должен храниться в LittleFS.

Для ESP32:

- NVS используется для небольших настроек и служебных параметров;
- буфер измерений должен храниться в LittleFS;
- при необходимости допускается использование NVS для компактных служебных данных.

SPIFFS рассматривается только как запасной или legacy-вариант, если LittleFS по техническим причинам нельзя использовать на выбранной плате или версии SDK.

Измерения не должны храниться в EEPROM как основной очереди, потому что объем EEPROM ограничен и она не подходит для накопления данных за несколько дней.

Локальный буфер измерений должен:

- сохранять измерения при отсутствии связи с сервером;
- хранить записи в порядке создания;
- отправлять накопленные записи на сервер после восстановления связи;
- удалять записи только после подтверждения успешной отправки;
- использовать максимально доступный объем выделенной памяти;
- иметь настраиваемый лимит размера или количества записей, если пользователь хочет ограничить буфер;
- при заполнении памяти удалять самые старые данные по дням и освобождать место для новых записей;
- защищаться от повреждения данных при перезагрузке или потере питания.

Требуется уточнить:

- проверить поддержку LittleFS на выбранных платах ESP8266 и ESP32;
- как резервировать место под прошивку, web-интерфейс и OTA, чтобы буфер измерений не занял всю файловую систему;
- нужен ли отдельный журнал ошибок устройства.

## 13. Безопасность

Требуется определить:

- авторизацию пользователей;
- авторизацию устройств;
- авторизацию локального web-интерфейса устройства;
- хранение паролей;
- хранение токенов устройств;
- защиту MQTT/API;
- разграничение доступа между пользователями;
- защиту персональных данных.

Локальный web-интерфейс устройства должен требовать пароль администратора устройства. Пароль не должен храниться в открытом виде. Внутри локального web-интерфейса роли не разделяются: авторизованный пользователь имеет доступ ко всем локальным функциям.

IoT-устройства должны авторизоваться через пару `deviceId` + `deviceToken`. Токен устройства не должен храниться на сервере в открытом виде. Устройство должно иметь доступ только к своим MQTT-топикам и своим командам.

## 14. Мультиязычность

Пользовательские интерфейсы системы должны поддерживать три языка:

- русский;
- украинский;
- английский.

Требования:

- пользователь должен иметь возможность выбрать язык интерфейса;
- выбранный язык должен сохраняться в профиле пользователя;
- web-интерфейс и мобильные приложения должны использовать единый набор переводов или синхронизированную структуру ключей;
- системные уведомления должны отправляться на выбранном пользователем языке;
- web-интерфейс системы и мобильные приложения должны поддерживать все три языка;
- локальный web-интерфейс устройства в MVP поддерживает русский язык;
- мультиязычность локального web-интерфейса устройства переносится после MVP, если это позволяет объем памяти устройства.

## 15. MVP

Предварительный MVP:

- одно устройство ESP отправляет данные на backend;
- минимальный локальный web-интерфейс устройства для настройки Wi-Fi, MQTT, модулей и локального обновления прошивки;
- локальный web-интерфейс устройства в MVP может быть одноязычным на русском языке;
- настройка пинов датчиков через локальный web-интерфейс устройства;
- калибровка весового модуля через локальный web-интерфейс устройства;
- backend принимает и сохраняет измерения;
- web-интерфейс показывает список ульев и карточку одного улья;
- график веса, температуры и влажности;
- роли MVP: владелец пасеки и администратор системы;
- базовые события: открытие улья и резкое изменение веса;
- измерение и отправка уровня батареи/питания устройства;
- мобильное приложение получает MVP-набор push-уведомлений или хотя бы отображает текущее состояние улья на раннем этапе;
- ручная привязка устройства к улью через админ-интерфейс;
- локальное обновление прошивки через web-интерфейс устройства;
- аварийное обновление/восстановление прошивки через USB/serial.

Команды от backend к устройству в MVP могут быть не реализованы, но архитектура MQTT-топиков и backend должна учитывать их появление после MVP.
