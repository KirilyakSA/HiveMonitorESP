# Техническое задание: web frontend HiveMonitor

## 1. Назначение

Frontend должен дать пасечнику рабочий web-интерфейс для ежедневного контроля пасеки:

- видеть организации и доступные пасеки;
- создавать пасеки и ульи;
- видеть новые IoT-устройства, появившиеся после настройки firmware;
- привязывать устройство к улью;
- смотреть текущие показатели улья;
- смотреть историю показателей;
- смотреть журнал событий пасеки и улья.

Frontend не является лендингом. Первый экран после входа - рабочая панель пасеки.

## 2. Пользователи и роли

Целевая модель ролей по ТЗ:

- роли организации: `organization_owner`, `organization_admin`, `organization_member`, `organization_viewer`;
- роли пасеки: `apiary_owner`, `apiary_admin`, `apiary_worker`, `apiary_viewer`.

MVP backend пока не отдает permission matrix в API, поэтому MVP frontend работает с текущим пользователем как с пользователем, которому backend уже разрешил доступ.

Когда backend добавит permissions, frontend должен:

- скрывать недоступные действия;
- показывать read-only состояние для viewer;
- запрещать destructive actions без подтверждения;
- не полагаться только на UI: backend остается источником прав.

## 3. Backend API MVP

Frontend использует текущие backend endpoints:

```text
POST /auth/register
POST /auth/login
GET  /me

GET  /organizations/
POST /organizations/

GET  /apiaries/?organization_id={organizationID}
POST /apiaries/
GET  /apiaries/{apiaryID}/hives
POST /apiaries/{apiaryID}/hives
GET  /apiaries/{apiaryID}/devices/unassigned
POST /apiaries/{apiaryID}/devices/{deviceUUID}/assign
GET  /apiaries/{apiaryID}/events
GET  /apiaries/{apiaryID}/advice
PATCH /apiaries/{apiaryID}/advice/{adviceCode}
GET  /apiaries/{apiaryID}/calendar/tasks
POST /apiaries/{apiaryID}/calendar/tasks
PATCH /apiaries/{apiaryID}/calendar/tasks/{taskID}

GET  /hives/{hiveID}/telemetry/latest
GET  /hives/{hiveID}/telemetry/history
GET  /hives/{hiveID}/events
```

JWT access token хранится в `localStorage` в MVP. После появления refresh tokens хранение и обновление сессии нужно пересмотреть.

## 4. Firmware/provisioning flow

Целевой flow:

1. Админ пасеки сбрасывает или впервые включает устройство.
2. Firmware поднимает локальную AP.
3. Пользователь вводит Wi-Fi, MQTT credentials пасеки и `apiaryId` в локальном web UI firmware.
4. Firmware публикует:

```text
apiaries/{apiaryId}/devices/{deviceId}/telemetry
apiaries/{apiaryId}/devices/{deviceId}/events
apiaries/{apiaryId}/devices/{deviceId}/status
```

5. Backend создает устройство как `unassigned` в этой пасеке.
6. Web показывает устройство в списке непривязанных устройств.
7. Админ выбирает улей и режим обработки старых данных:

```text
attach_to_hive
keep
delete
```

8. Backend создает активную привязку устройства к улью.

Legacy topics `hives/{deviceId}/...` остаются compatibility mode. Production UI должен считать apiary-aware provisioning основным путем.

## 5. MVP экраны

### 5.1 Auth

Функции:

- вход по email/password;
- регистрация нового пользователя;
- сохранение access token;
- выход.

Ошибки backend должны отображаться пользователю без технического stack trace.

### 5.2 Организации

Функции:

- список организаций текущего пользователя;
- создание организации;
- выбор активной организации.

### 5.3 Пасеки

Функции:

- отдельный пункт меню `Пасеки`;
- список пасек выбранной организации;
- основной статус пасеки;
- отдельный бейдж `Нужна привязка`, если на пасеке есть непривязанные устройства;
- карта с маркерами расположения пасек;
- выбор пасеки из списка или с карты;
- правая панель выбранной пасеки со сводкой и кнопкой `Перейти к пасеке`;
- создание пасеки через полноценную форму;
- поля пасеки:
  - название;
  - описание;
  - страна;
  - регион;
  - населенный пункт;
  - адрес;
  - описание места;
  - координаты;
  - часовой пояс.

Карта:

- MVP использует OpenStreetMap/Leaflet;
- в коде должна быть абстракция map provider;
- форма создания пасеки позволяет выбрать точку на карте кликом;
- при клике по карте выполняется reverse geocoding через OpenStreetMap/Nominatim;
- если reverse geocoding не сработал, координаты остаются заполненными, адресные поля пользователь вводит вручную.

Целевые будущие поля:

- weather provider;
- подключенная метеостанция;
- настройки уведомлений;
- настройки alert rules.

### 5.4 Ульи

Функции:

- список ульев пасеки;
- создание улья;
- выбор активного улья;
- отображение типа, номера, количества рамок и магазинов.

Целевые будущие поля:

- порода пчел;
- матка;
- статус семьи;
- пользовательские и системные теги;
- абстрактная позиция улья на схеме пасеки.

### 5.5 Непривязанные устройства

Функции:

- список устройств, которые backend видит в пасеке, но которые не привязаны к улью;
- привязка устройства к выбранному улью;
- выбор режима старых данных:
  - привязать к улью;
  - оставить диагностическими;
  - удалить.

UI не должен использовать простой `offline` для sleeping devices. Корректная модель связи - `last_telemetry_at`, `expected_next_telemetry_at`, `missed_telemetry_count`.

### 5.6 Карточка улья

Карточка улья должна показывать:

- вес;
- изменение веса;
- температуру;
- влажность;
- открытие улья;
- батарею;
- RSSI;
- время последнего измерения.

Если данных нет, UI должен явно показывать пустое состояние, а не нули.

### 5.7 История телеметрии

MVP:

- простой график одной метрики за последние 24 часа;
- автоматический выбор `weight`, если он есть, иначе `temperature`;
- latest values отдельными карточками.

Следующий инкремент:

- выбор метрики;
- диапазон дат;
- сравнение нескольких ульев;
- weather overlay.

### 5.8 Журнал событий

MVP:

- журнал пасеки;
- журнал выбранного улья;
- отображение типа события, сообщения и времени.

Следующие инкременты:

- ручное создание событий пользователем;
- комментарии и результаты работ;
- вложения;
- фильтры;
- связь событий с задачами, alert rules и системными тегами.

### 5.9 Советы и календарь работ

MVP:

- dashboard получает советы из `GET /apiaries/{apiaryID}/advice`;
- блок `Советы / Подсказки` показывает severity, заголовок, текст и действия;
- пользователь может скрыть или отложить совет;
- блок `События` объединяет device events, задачи календаря и сезонные рекомендации;
- вкладки: `Важные`, `Запланированные`, `Просроченные`, `Сезонные`;
- задачи можно отметить выполненными, отложить или скрыть.

Backend остается источником правил. Frontend не должен давать жестких ветеринарных назначений или дозировок, только безопасные подсказки в стиле “проверьте”, “запланируйте”, “обратите внимание”.

## 6. UX требования

- Интерфейс должен быть плотным, рабочим и удобным для регулярного использования.
- Не использовать маркетинговый hero или лендинг как первый экран.
- Главная рабочая единица - пасека и улей, не устройство.
- Пустые состояния должны быть понятными.
- Ошибки backend должны быть видимыми рядом с рабочим контекстом.
- Действия создания и привязки должны быть доступны без перехода в отдельные страницы.
- UI должен быть адаптивным для desktop/tablet/mobile.

## 7. Нефункциональные требования

- SPA на React + TypeScript.
- Dev server: Vite.
- API base URL через `VITE_API_BASE_URL`.
- Для локального dev Vite proxy на backend `localhost:8080`.
- Состояние MVP допускается хранить в React state без внешнего state manager.
- Код API должен быть отделен от UI.
- Не хранить MQTT passwords или firmware secrets в web frontend.
